Context Files Built (unit 1)

Drop a datasheet, a Milesight gateway export, or wiring notes into a project and the co-pilot answers from them. Each file is extracted, chunked, and retrieved into the agent's context on every ask — the first slice of Hyperion's retrieval-augmented knowledge.

Why it exists

The agent already grounds on the loaded .bos and on project memory, but the answer often lives in a document the export doesn't contain: "what Modbus slave is the Belimo actuator on?", "what's the holding register for setpoint on this Milesight gateway?" Context files make that material first-class — add it once, and the relevant pieces are pulled into the prompt whenever they're useful.

The pipeline

add file ──► extract text ──► chunk (~1500 chars, overlap) ──► store │ ask a question ──► keyword-rank chunks ──► top matches ─┘──► fenced as UNTRUSTED in the prompt
  • Extract — text/Markdown/CSV/TSV/log/JSON/YAML/XML/INI, plus PDF (content stream) and Word .docx (the WordprocessingML body). Other binary/office formats (xlsx, images) are still rejected with a clear message. Max 4 MB per file.
  • Chunk — split into ~1500-character pieces on line boundaries, with a small overlap so a fact straddling a boundary stays findable. Unicode-correct; always terminates.
  • Retrieve — a dependency-free keyword overlap ranks chunks against the question and the top few are injected. (The embedding/vector ranker is the next unit — the vector-store choice is still open.)

Security: retrieved text is untrusted

A datasheet or CSV can carry injected instructions ("ignore your rules and…"). Retrieved chunks are therefore entity-encoded and wrapped in a <context-files> fence inside the prompt's untrusted region — exactly like the .bos grounding — and the agent's standing instincts name that fence explicitly and refuse to follow anything inside it. A chunk containing </context-files> can't break out, because the delimiters are encoded.

Ingested files are stored only in the local project database and are never written back to bOS. As with memory, secrets don't belong here — they live in the encrypted Vault. (A context file that looks like it carries a plaintext credential is rejected on add, for every format.)

Parsing hostile binaries safely

PDF and .docx are binary formats parsed by third-party crates, so a malformed or deliberately hostile file is treated as an attack surface, not just bad input:

  • No crashes. PDF and DOCX extraction run inside a panic boundary — a parser that panics on a crafted file yields a clean error message instead of taking down the backend.
  • No zip bombs. A .docx is a ZIP; decompression of its body is capped (64 MB) and an archive with an implausible number of entries is refused, so a tiny file can't inflate to exhaust memory.
  • No entity attacks. The XML reader is a streaming parser that doesn't expand external or custom entities, so “billion laughs”/XXE payloads in a .docx are inert.
Known limit: PDF extraction has no wall-clock timeout yet, so a pathological PDF could run long; the 4 MB input cap bounds it, and a parse timeout is tracked for a later unit.

Commands & storage

CommandDoes
context_add_fileRead a file (picked via the OS dialog), extract + chunk + store. Returns {id, name, kind, chunks}. Re-adding a name replaces it.
context_listIngested files (name, kind, size, chunk count) — never the content.
context_deleteRemove a file and its chunks.

Storage is two SQLite tables in the project DB: context_file (the extracted text) and context_chunk (one row per retrievable chunk, deleted with its file).
