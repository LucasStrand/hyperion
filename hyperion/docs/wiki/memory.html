Project Memory Built

The co-pilot's long-term memory. Short, operator-authored notes that live with the project and are loaded into the agent's context on every ask — so it remembers the things that don't change between sessions.

Why it exists

The agent already grounds each answer in the loaded .bos (see The Agent). But a lot of what an integrator needs the co-pilot to know isn't in the export: "the main pump is on Modbus slave 3", "this site's KNX area is 1.1", "the customer wants the lobby scene to fade over 4 seconds". Without memory those facts must be re-typed into every conversation. Project Memory makes them durable: write a note once, and it is silently prepended to the agent's system prompt from then on.

What a note is

Each note is a small typed record in the project's SQLite store (the forward-compatible memory table reserved back in Phase 0). One row per slug — saving the same slug again updates in place rather than piling up duplicates.

memory row = id · mtype · slug · body · updated_at │ │ │ └ the note text the agent reads │ │ └ short name / key (slugified); upsert key │ └ category (one of four, below) └ stable row id (used for delete)
TypeFor
projectFacts about this install — addressing, gateways, who the customer is.
featureHow a built feature works, so the agent stays consistent with it.
referenceExternal facts — a datasheet value, a Milesight register, a KNX DPT.
securitySecurity-relevant reminders — never the secret itself (that lives in the Vault).

The type is validated in the backend: anything outside project | feature | reference | security is rejected. The slug is normalized with the same slugify the project store uses, and an empty body is refused.

How it reaches the agent

On every agent_ask, the active project's notes are read and folded into the system prompt as their own clearly-labeled section — ahead of the untrusted <bos-data> fence, because memory is written by the operator and is therefore trusted context:

INSTINCTS (system prompt) │ ├─ # Project memory (operator-authored, trusted) │ - [project] Main pump is Modbus slave 3. │ - [reference] Lobby KNX area = 1.1 │ └─ # Loaded system context (untrusted data) <bos-data> … the loaded .bos grounding … </bos-data> ▼ active runtime → text answer

The block is token-bounded (capped at ~4 KB, truncated with a marker if exceeded) so a long note can never crowd out the question or the grounding. When no project is open, or a project has no notes, the section is simply omitted — an ask still works.

Secrets never go here. Memory is loaded verbatim into the prompt, so a password or token written into a note would become model context. Credentials belong only in the encrypted Vault; the grounding builder never reads the vault, and the UI says so.

The UI

Memory lives where you talk to the agent — a collapsible Project memory panel at the top of the 💬 co-pilot dock. Expand it to see the saved notes (type chip · name · body), delete any with a click, or add a new one with the type selector, a short name, and the note text. Every rendered value is HTML-escaped, the same discipline the rest of the dock follows.

Commands (Tauri ⇄ webview)

All three operate on the active project's database and error clearly (no project is open — create or open one first) when none is, mirroring import_bos.

Boundaries

  • Strictly local — notes live in the project's git-ignored SQLite store; nothing is synced.
  • Read-only toward bOS — memory is context for the agent, never written back to the system.
  • No secrets — the vault is the only home for credentials; memory is plain context.
