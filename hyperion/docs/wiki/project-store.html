The Project Store

How Hyperion persists your work: each project is a folder with a local SQLite database holding ordered .bos snapshots — the renderer reads the active snapshot.

Read-only toward bOS SQLite (bundled) Rust core · src-tauri/src/projects.rs 5 Tauri commands

What it is

Up to Phase 0 the app rendered a single, pre-parsed configuration. The project store turns Hyperion into a multi-project workbench: you create a project, import one or more .bos exports into it, and each import is saved as an ordered snapshot. Opening a project reloads its most recent (or pinned) snapshot into the Configurator view — exactly the tree, grouped settings and playbook grading you already know.

Still strictly read-only. The store only ever reads your exported .bos and writes its own local copy into the project database. Nothing is ever written back to a live bOS Server — the trusted model is preserved.

The parser runs once per import; its JSON is captured straight into the database (snapshot.map_json), so re-opening a project never re-parses and works offline.

Using it

  1. Click New in the app bar and name the project (e.g. GB1625 Building).
  2. The project opens automatically and Import .bos… becomes enabled.
  3. Pick a .bos export — it is parsed, saved as a snapshot, and rendered.
  4. Re-open any project later from the Project dropdown; the count in parentheses is its snapshot total.

Security & safety notes

Project ids are validated against path traversal. Folder names come from slugify ([a-z0-9-] only) and open() rejects any id that is absolute or contains /, \ or .. — a project can never reach outside the projects root.
  • Local only. project.db and the hyperion-projects/ tree are git-ignored; they never leave the machine.
  • No silent failures. Every fallible step returns a typed error surfaced to the UI; mutex poisoning is recovered rather than cascaded.
  • Forward-compatible. Opening an older project self-heals missing tables without ever downgrading a newer schema_version.
