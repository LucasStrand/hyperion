Agent Roster Built

The co-pilot isn't one monolith — it's a roster of role-specialized agents. A built-in router sends each question to the right expert (a Coordinator steps in for general or unmatched questions); every expert carries its own standing instincts, and an explicit share protocol says when to hand work to a colleague.

Why a roster

Building a bOS system spans very different skills: wiring Modbus/KNX objects in the Configurator, tuning schedules and trends in the Service, designing a client UI that delights a customer, hunting documentation, enforcing standards, and keeping secrets safe. One generic prompt does all of them adequately and none of them well. A roster lets each answer come from an agent whose instincts are tuned for that exact job — while a single Coordinator keeps the experience seamless: you just ask, and the right specialist replies.

How routing works

Pick an agent from the who answers dropdown in the chat dock, or leave it on Auto and the router picks the specialist for you (the Coordinator handles anything unmatched). Routing is deterministic and offline — no model call, so it's instant, free, and unit-tested. Each specialist is scored by whole-word keyword hits in your question (a focused node's type adds a nudge by substring, so selecting a ModbusRegister tips an ambiguous question toward the Configurator-Expert). The highest scorer wins; ties break by roster order; if nothing scores, the Coordinator takes it.

ask + (focused node type) │ ▼ route() ──score specialists by keyword/type──► best match │ │ nothing scores ▼ │ Configurator-Expert (e.g.) ▼ │ Coordinator ◄───────────────────────────── answer + suggested handoffs

Every answer is labelled with the agent that produced it and whether the Coordinator auto-routed it, plus chips for any handoffs the share protocol suggests — e.g. a Configurator question that mentions a plaintext password surfaces a Security chip.

The share protocol

Each agent declares a small set of handoff rules: when keyword X appears, this is really Y's job. It's data, not guesswork — every handoff target is a real roster member (checked by a test), and the suggestions are shown alongside the answer so you can re-ask the right expert. This keeps each agent honest about the edges of its competence instead of bluffing past them.

Instincts — versioned & yours to shape

Every agent ships with built-in instincts: a short block of standing heuristics injected into its system prompt (the Configurator-Expert always cites real node paths and emits a runnable playbook; the Security agent leads with the security reflex; and so on). You can refine any agent's instincts per project in the Agents & instincts panel.

  • Append-only history. Each save is a new version. Nothing is overwritten.
  • Non-destructive revert. “Revert to v2” (or to the built-in default) copies that body forward as a new version — your later edits are never lost.
  • Guarded on write. Bodies are length-capped (8 KB) and scanned for plaintext secrets; a pasted key is refused and pointed at the Vault.
  • Trusted, but bounded. Custom instincts are your configuration, so they sit in the trusted-instructions region of the prompt — framed as refinements that never override the standing security-reflex and read-only rules.
Instincts and history are stored per project in the SQLite agent_instincts table (one row per (agent_id, version)). Version 0 is the built-in baseline and lives in the binary, not the database, so a fresh project starts from sensible defaults with zero rows.

Security posture

The roster changes who answers, never what the app may do. Every agent stays strictly read-only toward bOS. Routing on an untrusted node type only ever selects a benign, trusted agent — it can inject nothing. Custom instincts can refine tone and focus but cannot disable the standing security-reflex or read-only guarantees, and the untrusted .bos grounding stays fenced and entity-encoded exactly as before. Secrets remain only in the encrypted Vault.
