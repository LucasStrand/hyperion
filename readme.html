bOS Co-pilot — guide

A local, read-only assistant for implementing features in ComfortClick bOS.
Read-only · never writes to your bOS system · you apply every change yourself

·What it is

bOS Co-pilot loads your exported .bos config, shows your real object tree (the same tree the Configurator shows), and walks you step-by-step through a feature — each step pinned to the actual node you need to touch.

The part that makes it more than a checklist: it reads your .bos and grades every step against it. Each step is badged ✓ Implemented◐ Partial○ To-do automatically, so you can see at a glance what is already in your system and what is actual work left.

Core assumption: the .bos you load is your system. What the tool sees in the export is what you see in the Configurator. So a green ✓ Implemented badge means it is already there — you do not re-open the Configurator to confirm it. You read the badge and move on to the To-do steps.

It is a guide, not an editor. It reads the export and renders instructions and status. Every change is applied by you, in the Configurator. Nothing it does can reach or modify your live system.

1Install & run

You need Python 3, plus two packages (one-time):

pip install nrbf flask

Then, from the folder that contains bos_copilot.py, bos_explore.py and your export, launch it pointed at your config:

python bos_copilot.py GB1625-Building.bos

It parses the file (the first run is the slow one — it caches the result to bos_map.json so later launches are instant), starts a local server at http://127.0.0.1:5000, and opens your browser automatically.

Keep the terminal window open while you use it — that's the server. Press Ctrl+C there to stop. The tool always reflects the .bos you launched it on; after you apply changes in the Configurator and re-export, the cache rebuilds on the next run (it re-parses whenever the .bos is newer than bos_map.json), so the badges re-grade against the new export.

2The layout

Two panes, plus a floating Guide panel that appears when you pick a feature.

Left — Object tree

Your real config tree. Type in the search box to highlight matching nodes. Click any node to inspect it in the panel on the right.

Right — Inspector

Tabs Settings · Values · Functions · Usages · Notes · Info. Shows the node's type and settings; for a Program it reconstructs the Triggers and the ordered Commands tree (If / SetValue / Delay / Run / Comment). It also shows what the node reads, what it writes, and (Usages) what references it.

Floating — Guide

Pick a feature from the dropdown (top bar) and a panel docks bottom-right: the ordered steps, each with its live status badge. Click a step to highlight its target node (yellow) in the tree.

3Reading a node (left pane)

  1. Search or expand to the node, then click its name.
  2. The inspector on the right shows everything the parser resolved for it.

Example — clicking OR Ventiler På status shows type=Gate, Type=1 (1 = OR, 0 = AND), and its 5 machine-valve inputs. That's how you confirm bindings straight from the export — and since the export is your system, that reading is authoritative.

4Using a guide

1
Pick a feature in the Guide dropdown (top bar). The panel docks bottom-right.
2
Read the summary, then work top-to-bottom. Each step has an action badge: navigate go to a node   add create a node   set set a value / binding.
3
Each step also shows a status badge graded against your .bos: ✓ Implemented◐ Partial○ To-do Skip the green ones — they're already in your system. Do the To-do (and finish the Partial) steps.
4
Click a step → its target node lights up yellow in the left tree. Open the same node in your Configurator and do what the detail line says. Steps that change a Program show a Before / After diff of the command tree (preview only).

Two guides ship with the app: Verify roof pump follows any machine pump (takpump) walks the roof-cooling chain, and Stoppa takpump vid läckage builds a leak interlock — the latter shows the status grading in action, with the wiring it depends on already badged green and only the new Program badged To-do.

5How the status grading works

This is the heart of the tool, and the reason you don't hand-verify anything. When you load a guide, the app re-reads each step's target from the parsed .bos and evaluates the step's check conditions. The header shows a running X/Y implemented count, and each step gets:

BadgeMeaning
✓ ImplementedEvery condition for this step is already true in your .bos. Nothing to do.
◐ PartialSome conditions are met, some aren't. Finish the missing ones.
○ To-doNone of the conditions are met yet. This is real work.

Under the badge, each individual condition is listed with ✓ or ✗ so you see exactly which part is present and which is missing — e.g. "✓ Program exists" but "✗ trigger on Läckage utlöst".

Because the .bos the tool reads is your system, these badges are the verification. A condition shown ✓ is in place; you don't go confirm it by hand. Your job is to clear the ✗ items, re-export, and watch them turn green.

6The workflow loop (how new features get built)

The "intelligence" — turning a plain-language feature into exact steps against your real objects, with check conditions so the app can grade them — happens in chat with me. The app renders it. So the loop is:

  1. You describe the feature you want.
  2. I write a playbook JSON grounded in your real node paths, with a check on each step so the app auto-grades it, and I hand it to you.
  3. You drop the file into the playbooks/ folder (next to bos_copilot.py).
  4. Refresh the app — it appears in the dropdown, walks you through it, and tells you which steps are already done in your .bos.

Playbook file format

Plain JSON, one file per feature. The fields the app actually renders are below (the reference copy lives in playbooks/_SCHEMA.json):

{
  "feature": "Short name shown in the dropdown",
  "summary": "What this change does and the rule it implements.",
  "config":  "GB1625-Building.bos",
  "steps": [
    {
      "n": 1,
      "title":  "Short imperative step title",
      "action": "navigate | add | set",
      "target": "Tasks\\Exact\\Node\\Path",
      "detail": "Exactly what to click/do in the Configurator.",
      "check": {
        "node": "Tasks\\Exact\\Node\\Path",
        "all": [
          {"exists": true, "label": "the node exists"},
          {"trigger": "Läckage utlöst", "label": "trigger on the leak flag"}
        ]
      },
      "diff": {
        "node": "Tasks\\Exact\\Node\\Path",
        "section": "commands",
        "before": [],
        "after": [ {"cmd": "SetValue", "text": "...Spärra.Value = True", "added": true} ]
      }
    }
  ]
}
FieldMeaning
actionSets the action badge: navigateaddset
targetA real node path. Clicking the step highlights it in the tree. Double backslashes in JSON (\\).
checkWhat makes the step auto-grade Implemented / Partial / To-do. node is read from your .bos, and every predicate in all is evaluated against it.
diffOptional Before / After command-tree preview. Mark added lines "added": true and removed lines "removed": true. Preview only.

check predicates

Each entry in check.all is one condition tested against the check.node, with a label shown in the ✓/✗ list:

PredicateTrue when…
{"exists": true}the node is present in the .bos.
{"referencedBy": "Name"}some object whose path contains Name references this node (its Usages).
{"writes": "Name"}this node writes to an object whose path contains Name.
{"trigger": "Text"}(Program) a trigger contains Text.
{"command": "Text", "under": "Parent"}(Program) a command contains Text; optional under requires it nested under a command containing Parent.

Files whose name starts with _ (like _SCHEMA.json) are ignored by the app. A legacy verify field may appear in older playbooks; the app does not display it — the check block replaces it.
