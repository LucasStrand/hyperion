The Agent & Runtime Adapter Built

The brain Hyperion talks to. One small interface over three interchangeable backends — preferring the model running on your own machine — grounded in the loaded .bos so its answers cite real nodes, not guesses.

Why it exists

Hyperion is meant to be operated by an AI agent that knows ComfortClick bOS and can teach, not just answer. But which model? The two operators run capable hardware, so Hyperion prefers a local runtime and only reaches the cloud when it has to. The adapter makes the model a swappable part: the rest of the app talks to one ask() and never cares who answered.

The three runtimes

Detected and selected in strict preference order — the first one available wins, unless you override it:

OrderRuntimeHow it's reachedDetected by
1Claude Codeclaude -p, prompt on stdin the claude executable on PATH
2Codexcodex exec, prompt on stdin the codex executable on PATH
3OpenRouterHTTPS chat/completions (cloud fallback) a key in OPENROUTER_API_KEY or the unlocked vault

Discovery is Windows-aware: it honours PATHEXT, so an npm claude.cmd shim and a native claude.exe are both found, and batch shims are launched through cmd /C (which CreateProcess can't do directly). Override the choice with the HYPERION_AGENT_RUNTIME environment variable (claude / codex / openrouter); the OpenRouter model defaults to a sensible Claude slug and is overridable via HYPERION_OPENROUTER_MODEL.

How an answer is built

your question + focused node (optional) │ ▼ INSTINCTS (system prompt) + grounding from the LOADED .bos │ ├─ config name + object count │ ├─ top-level sections (name × children) │ ├─ object types present (with counts) │ └─ the node you're looking at (writes / refs) ▼ active runtime → Claude Code | Codex (prompt on stdin, no shell) └ or OpenRouter (Bearer key, never logged) │ ▼ text answer → chat dock (```playbook fence → "Load playbook" → guide engine)

The grounding is token-bounded and built fresh from the in-memory render store on every ask. It is the bridge that makes the loaded .bos the agent's source of truth — the same principle the Configurator already lives by. Phase 3's retrieval (embedded context files) will be prepended at exactly this seam.

Instincts

Every request is prefixed with a standing system prompt — the agent's instincts (milestone M5). They make good behaviour the default, not a matter of luck:

  • Teach, don't just do — state what to do, show pedagogical steps, give finished code, and when the change is a concrete edit sequence, emit a runnable ```playbook JSON block.
  • Ground every claim in the loaded system; cite real node paths, never invent nodes.
  • Ask when uncertain, and request context (a datasheet, the Milesight export) when it would genuinely help — naming exactly what and why.
  • Stay read-only toward bOS; the user applies edits in the real Configurator.
  • Security reflex — flag a plaintext password/key, recommend the vault, never repeat a secret in full.
  • Right artifact for the shape — a diff, a flowchart, a table, a timeline.
  • Surface eurekas — make the case for a worthwhile improvement and show how.

What's safe about it

  • No shell, ever. Local CLIs are spawned by exact executable with constant arguments; your question travels as stdin data, so it can never be read as a flag or shell syntax.
  • No wedged processes. stdin is written and stdout/stderr drained on dedicated threads (deadlock-free), under a hard 180-second timeout that kills a hung child.
  • The key stays secret. The OpenRouter key is read from the environment or the unlocked vault, sent only as a Bearer header, and never written to an error or log.
  • The vault is never read into a prompt. Grounding draws only on the .bos tree — secrets never become model context.
  • The dock can't be injected. Model output is HTML-escaped before rendering; only the fenced-code structure becomes markup.
Trust boundary. Detecting a CLI on PATH means Hyperion will run whatever claude/codex is first on PATH — the same trust you extend by running those tools yourself. Acceptable for a single-operator desktop tool; revisit if Hyperion ever runs untrusted profiles.

Playbook hand-off Built

When the agent emits a runnable ```playbook block, the chat dock now does more than tag it. Beside the ▶ Playbook label sits a ▶ Load playbook button. Clicking it parses the block, validates it (a feature name and a non-empty steps array), and routes the parsed object straight into the same Configurator guide engine the bundled playbooks/*.json files use — through loadGuideFromObject() instead of fetching a file by name.

From there it is indistinguishable from a shipped playbook: the steps render in the guide panel, each step's check is auto-graded against the loaded .bos (✓ Implemented / ◐ Partial / ○ To-do), targets highlight in the tree, and Before/After diffs preview inline — all still strictly read-only toward bOS. Malformed JSON or a missing feature/steps never throws; it surfaces a clear inline error under the block.

The agent invents the plan; the engine supplies the proof. Because the grading runs against your live system, a green step is already in place — you don't re-verify it by hand.
