The Box-Scan Verifiering

· A Charm for Placing the Right Crate ·

How a barcode scanner, a Windows herald, and the bOS enchantment work together to tell the worker — in green or red — whether the right plastic box has landed in the right spot.

✦ ❧ ✦

The tale in one breath

Three boxes catch falling plastic at three positions. To prove each box sits where it belongs, a worker scans two barcodes with one handheld gun — first the spot, then the box. The scan station (a tiny Windows app from your scanner-to-comfortclick repo) whispers each code to the bOS server. A bOS Program compares the pair against the fixed pact — Plats 1 = BOX-A, Plats 2 = BOX-B, Plats 3 = BOX-C — and lights that spot green (right) or red (wrong) on the Inject visual page, with a banner and a beep.

IIWhat Must Be True Before Any of It Works

Prerequisites, grouped by where they live. None are exotic — but if one is missing, the spot simply never lights up.

⌨ At the scan station (PC)

  • A Windows PC that stays powered on by the boxes, running ScannerToComfortclick.exe.
  • .NET 8 Desktop Runtime installed.
  • One (or more) USB barcode scanner in HID-keyboard mode.
  • The scanner must send an Enter / carriage-return suffix after each code — that suffix is what triggers the send. (Most guns do by default; if not, scan the “add CR suffix” config barcode from the gun’s manual.)
  • A clear network route from this PC to the bOS server IP.

🏰 On the bOS server

  • RPC API enabled (Building → API → RPC service) with a username + password.
  • The string variable Devices\Scanner exists. already in your config
  • The Program Tasks\Box Scan Kontroll\Verifiera box and its variables + visual tiles, built from the co-pilot playbook. to build
  • Printed labels in place: SPOT-1/2/3 on the three positions, BOX-A/B/C on the boxes.

IIIThe Scanner Software — Do You Need to Change the Code?

The short answer

No source changes are required in the scanner-to-comfortclick repo. The two-barcode feature is pure configuration on the app side plus logic on the bOS side. The app already does exactly what we need: it sends each scanned string, verbatim, to one bOS variable.

How the app actually sends a scan

Reading the repo (ScannerListener.cs → ComfortclickClient.cs), the flow is:

  1. Raw-Input collects keystrokes per device into a buffer (so only your whitelisted gun is heard).
  2. On the Enter key it trims the buffer and, if it meets MinBarcodeLength, fires a send.
  3. It POSTs this JSON to /API/RPC/SetValue (Basic-auth):
{ "objectName": "Devices\\Scanner", "valueName": "Value", "value": "SPOT-1" }

So Devices\Scanner.Value becomes "SPOT-1", then "BOX-A", and so on — one variable, every scan. That is the whole reason our bOS Program tells spot-codes from box-codes by their prefix, not by which variable changed.

⚠ The one true gotcha — and its cure

The Program is triggered by Devices\Scanner.Value OnChange. If the same code is scanned twice in a row, the value does not change — so OnChange never fires and the second scan is silently ignored. In the normal spot→box→spot→box rhythm the value always alternates, so it usually works… until a worker re-scans the same spot, or two placements repeat a code.

The cure (bOS side, no repo change): make the very last command of Verifiera box: Set Value  Devices\Scanner.Value = "". Clearing it after every scan means the next code — even an identical one — is always a real change, so it always fires. (Clearing to "" re-fires once with an empty value, matches no If, then "" → "" is no change, so it stops. No loop.)

Avoid the tempting alternative of appending a timestamp/counter in the app — that would break the exact = "SPOT-1" matching the Program relies on.

VThe Window the Worker Watches

On the Inject visual page, each of the three positions gets a status tile (a BOSTheme.Controls.Boolean like your existing 84001–84005 tiles), placed where the box physically sits. A wide banner up top spells out the words.

VISetup, Spell by Spell

  1. bOS — open the gates. Building → API → enable the RPC service; set a username & password; note the server IP.
  2. bOS — build the logic. In the co-pilot, run the “Box-scan kontroll” guide: create the Box Scan Kontroll folder, its variables, and the Verifiera box Program (trigger Devices\Scanner.Value OnChange). Add the final Devices\Scanner.Value = "" clear (the gotcha cure).
  3. bOS — dress the window. On Inject visual, add the OK/Fel tiles per spot and the Meddelande banner; bind colours to the variables.
  4. Labels — print & stick. SPOT-1/2/3 on the three positions, BOX-A/B/C on the boxes.
  5. Scan station — point it home. In the app’s Settings, set Object Path Devices/Scanner, Value Name Value, and the RPC credentials. Whitelist the gun on the Scanners tab.
  6. Prove it. Test Connection → Start Listening → scan a pair (see below).

VIIThe Proving — and When It Sulks

A clean test run

Scan SPOT-1 → banner reads “Plats 1: skanna lådan”. Scan BOX-A → Plats 1 turns green. Now scan SPOT-2 then BOX-A (deliberately wrong) → Plats 2 turns red and beeps. No physical gun handy? In the Configurator you can type a value straight into Devices\Scanner.Value to simulate a scan.
