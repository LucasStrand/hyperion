The Encrypted Vault Built

Where Hyperion keeps the secrets it must never leak: bOS logins, device credentials, API tokens. Encrypted at rest, decryptable only on this machine, and never echoed back to the UI.

Why it exists

Hyperion catalogs real building systems — and to talk to them it needs real credentials. Those can never sit in a project's SQLite store, a snapshot, a note, or the wiki in plaintext. The vault is the one place a secret is allowed to live, and even there it is encrypted. A companion guardrail scans free text for credentials before they are stored as context, so a pasted password is caught instead of silently persisted.

How it is protected

One file on disk, vault.bin, holds every secret as a single authenticated-encryption blob:

vault.bin = nonce(12 bytes) || AES-256-GCM ciphertext ciphertext decrypts to = JSON { "<secret-name>": "<secret-value>", ... }
  • Cipher: AES-256-GCM (authenticated — tampering is detected, not silently decrypted).
  • Nonce: a fresh 96-bit random nonce per save, from the OS CSPRNG (OsRng).
  • The key (DEK): a 32-byte data-encryption key that never touches disk in the clear. It lives in the OS keychain (Windows Credential Manager via the keyring crate; service hyperion-vault, account dek, stored base64).
  • In memory: the DEK is held only while the vault is unlocked, and is zeroized on lock and on drop — along with the transient base64/byte buffers used to load it.
At rest the blob is useless without the keychain DEK; the keychain entry is itself protected by the operating-system user login. Two independent locks, both local to the machine.

Locking & unlocking

Unlock ─▶ fetch DEK from OS keychain (created on first use) ─▶ if vault.bin exists, verify the DEK actually decrypts it ─▶ hold DEK in memory; create an empty vault on first run Lock ─▶ zeroize the in-memory DEK (the blob stays encrypted on disk)
Coming in Phase 1 step 2 — Microsoft Entra SSO. Unlock will be gated behind a successful Microsoft Entra sign-in (OAuth 2.0 authorization-code + PKCE, loopback redirect http://localhost) for defense-in-depth. Until that lands, the OS keychain is the gate.

The plaintext-secret guardrail

Before any free text is stored as project context, scan_secret checks it for things that belong in the vault. Every finding is reported with a masked sample, so the warning itself never echoes the secret:

DetectorCatches
private_keyPEM private-key blocks (-----BEGIN … PRIVATE KEY-----)
aws_access_keyAWS access-key IDs (AKIA…)
bearer_tokenBearer … / Authorization: tokens
credential_assignmentpassword=, secret:, api_key=, token= … with a non-trivial value

Boundaries

  • The vault is strictly local — git-ignored (vault.bin), per-machine, never synced.
  • Hyperion stays read-only toward bOS; the vault holds credentials, it does not push them anywhere.
  • Error messages never include raw secret material.
Tracked hardening (Phase 1 follow-up): secret values held transiently in memory during a get/set are not yet zeroized (the at-rest blob and the DEK already are). Wrapping values in Zeroizing/SecretString is a logged item.
